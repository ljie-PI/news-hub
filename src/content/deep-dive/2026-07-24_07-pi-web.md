## 一、定位与痛点

Pi Web 是为 [pi coding agent](https://github.com/badlogic/pi-mono) 提供的本地 Web 界面。纯 CLI 使用编程 Agent 时，历史会话散落在终端与 session 文件路径中，难以回溯；模型配置、技能开关、项目文件预览都得在终端里手动折腾。Pi Web 读取本地 pi 的 session 文件，把这一切搬进浏览器工作区：会话浏览、实时对话、模型配置、技能管理与项目文件预览一站式完成。

## 二、核心架构与技术细节

项目以 TypeScript 为主（约 92%），基于 Next.js（App Router）构建。后端 `app/api/` 下按职责拆分：`agent/` 创建并驱动 AgentSession 并通过 SSE 推送事件、`sessions/` 负责会话读取/重命名/HTML 导出、`models-config/` 读写 `models.json`、`skills/` 处理技能安装与启停、`files/` 提供文件列举与预览。前端 `AppShell`、`ChatWindow`、`MessageView`、`FileViewer` 等组件支持消息流、思考过程、工具调用渲染以及源码/diff/图片/音频/PDF 预览。默认读取 `~/.pi/agent/sessions` 下的 `.jsonl` 文件，支持 Git worktree 切换、会话 fork 与「从此处编辑」分支。

## 三、竞品对比与生态站位

相比 pi 原生 CLI，Pi Web 提供了结构化工具调用、可读 Markdown 与更清晰的会话浏览体验。它并非独立 Agent，而是 pi-mono 生态的可视化前端，定位类似 Open WebUI 之于本地大模型——降低配置门槛、增强可观测性（上下文用量、成本、压缩状态、system prompt 均可见）。

## 四、开发者反馈与局限性

项目约 2.1k star、309 fork、321 次提交、18 个发行版（自报口径），并提供中文 README，迭代活跃。局限在于强依赖 pi coding agent，独立使用价值有限；官方提示本地开发时避免运行 `next build` 以免干扰 dev server，工程约束仍有粗糙处。
