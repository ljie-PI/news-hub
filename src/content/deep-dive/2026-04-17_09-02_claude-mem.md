# claude-mem 深度报告

> ⭐ 59,751 stars (+1,897 today) | 作者：Alex Newman (@thedotmack) | 版本：v6.5.0 | License: AGPL-3.0

---

## 定位与痛点剖析

Claude Code 每次启动都是全新会话——之前做的事情、发现的 bug、定下的约定，全部清零。开发者不得不手动复述上下文，或维护一份越来越长的 CLAUDE.md。

**claude-mem** 的核心定位是：**Claude Code 的持久记忆层**。它作为 Claude Code 插件运行，全程自动——你不需要手动记录任何东西。

痛点场景：
- 上次调试发现了某个接口的奇怪行为，下次会话 Claude 完全不记得
- 项目约定（比如"用 spaces 不用 tabs"）每次都要重新说
- 多天跨越的长期任务，Claude 每次重连都需要热身
- 代码库越来越大，每次会话的 token 预算都浪费在"重新了解项目"上

---

## 核心架构与技术细节

claude-mem 是一个完整的工程系统，**不只是一个脚本**：

**6 个生命周期 Hook：**
- `SessionStart`：注入上次会话的压缩记忆
- `UserPromptSubmit`：捕获用户意图
- `PostToolUse`：记录每次工具调用的观察（这是信息密度最高的来源）
- `Stop`：会话暂停时处理
- `SessionEnd`：会话结束时生成摘要

**Worker Service（端口 37777）：**
- HTTP API + Web Viewer UI（可实时查看记忆流）
- 由 Bun 管理，独立进程
- SQLite 存储 sessions / observations / summaries
- Chroma 向量数据库支持语义检索

**3-Layer MCP 检索模式（~10x token 节省）：**
1. `search` → 获取 compact index（~50-100 tokens/result）
2. `timeline` → 获取时序上下文
3. `get_observations` → 仅为过滤后的 ID 获取完整内容（~500-1000 tokens/result）

**安装只需一行：**
```bash
npx claude-mem install
```
也支持 Gemini CLI、OpenCode 等其他 IDE。

## 竞品对比与生态站位

| 方案 | 持久化方式 | 自动化程度 | 检索方式 |
|------|----------|----------|--------|
| 手动维护 CLAUDE.md | 手动 | ❌ | 无 |
| `/memory` 命令 | 半手动 | 半自动 | 无 |
| **claude-mem** | **全自动** | **✅ hook 驱动** | **语义+关键词混合** |
| mem0 | API 调用 | 半自动 | 向量检索 |

claude-mem 的差异化在于**完全零干预**：你只需要正常使用 Claude Code，记忆系统在后台运行。不需要你发指令，不需要你复制粘贴。

值得注意的是项目已有配套的 $CMEM Solana 代币——这是一个带有 crypto 激励机制的 OSS 项目，目标是构建"实时 Agent 数据"生态。这在技术社区引发了一些讨论（"为什么一个开发工具需要币？"）。

## 开发者反馈与局限性

**正面评价：**
- Augment Code 专文报道，称其为"最实用的跨会话记忆方案"
- 支持 30+ 语言版本（README），国际化程度极高
- 社区活跃，Discord 服务器专门建立
- Web Viewer UI 让记忆内容可视化，调试体验好
- `<private>` 标签保护敏感内容不被存储

**局限性与争议：**
- AGPL-3.0 License：在网络服务器上部署修改版时须开源，商业使用需谨慎
- 部分核心模块（`ragtime/`）采用 PolyForm Noncommercial License，不可商用
- 依赖 Bun、Python(uv)、SQLite、Chroma，系统依赖较重
- 与 $CMEM 代币的关联引发社区对项目商业化意图的质疑
- Worker service 在端口 37777 常驻运行，对轻量环境有一定负担
- 目前主要优化针对 Claude Code，对其他 IDE 的支持程度有差异
