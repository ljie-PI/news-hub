# Unabyss：面向 AI 的 MCP 原生自更新上下文层

## 一句话定位

**MCP-native self-updating context layer for your AI** —— 一个基于 Model Context Protocol、能自动保持上下文最新的"AI 记忆 / 上下文中间件"。

## 背景：为什么 2026 年都在做 "Context Layer"

LLM 应用最大的两个工程问题始终是：(1) 上下文从哪里来、(2) 怎么保持新鲜。2025 年下半年 Anthropic 推出的 **MCP（Model Context Protocol）** 已成为事实标准，让 LLM 与外部数据源 / 工具的对接方式从私有 plugin 走向开放协议。围绕 MCP 的生态正在快速分层：底层是 servers（GitHub、Notion、Slack 等），中间层是 context router / memory，应用层是 Claude Desktop、Cursor、ChatGPT 等 client。Unabyss 切的就是中间层。

## 结构 B：产品形态推断

1. **MCP-native**：作为 MCP server 暴露能力，任何兼容 client（Claude Desktop、Cursor、Cline、Continue 等）都能即插即用。
2. **Self-updating**：核心卖点。意味着它会主动 pull / watch 用户的数据源（文档、代码库、Slack 消息、Notion、邮件等），并在后台增量索引、去重、摘要、向量化，保证 LLM 每次取用的都是最新版本——这解决了 RAG 系统最大的"陈旧索引"问题。
3. **Context layer**：在 LLM 请求时，根据 query 动态选择 / 压缩相关上下文，避免 token 浪费。可能内置 reranker、context pruning、引用追踪。

## 差异化与竞争

- 对手有：Mem0、Letta（前 MemGPT）、Zep、Cognee、Supermemory 等"AI memory"创业公司。
- Unabyss 的差异点压在 **MCP 原生** 与 **自动更新** 两个关键词。前者保证生态兼容性，后者解决工程实操痛点。
- 风险：MCP 生态本身在快速演进，server 标准、auth、权限模型仍在变化，过早押注某种实现可能需要频繁重写。

## 适用场景

- 个人知识工作者：把 Notion + Gmail + GitHub 接入，让 Claude 永远拥有最新上下文。
- 团队：作为共享 context layer，把内部 wiki / 工单 / 代码连入 AI 助手。
- 开发者：替代手写 RAG pipeline，专注业务逻辑。

## 价值评估

"Context layer" 是 2026 年 AI 基础设施最确定的赛道之一，MCP 又是当下最热的协议，Unabyss 的定位踩在两个红利点上。能否跑出来取决于：(1) 自动同步的稳定性与延迟；(2) 权限 / 隐私模型是否企业级；(3) 与主流 MCP client 的兼容深度。值得 PH 关注者收藏跟进。
