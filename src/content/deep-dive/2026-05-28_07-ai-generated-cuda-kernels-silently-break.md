## 背景

随着大模型时代算子库不断膨胀，业界开始尝试用 AI 自动生成 CUDA kernel，以期榨干 GPU 的最后一滴性能。NVIDIA 上个月发布了 **SOL-ExecBench**，一个包含 235 个产线级 CUDA kernel 的基准集，样本直接抽自 DeepSeek、Qwen、Gemma 与 Kimi 等主流模型训练栈。原帖作者所在团队将该榜单上若干\"AI 生成\"的高分提交移植到真实生产负载中验证，结果发现：很多 kernel 看似正确、跑分领先，**实际却以静默的方式破坏训练或推理流程**。

## 核心发现

帖中重点案例是 **fused embedding-gradient + RMSNorm 反向 kernel**——它出现在每一步 Transformer 训练末端，几乎决定了端到端吞吐。团队把基准里最快的一个生成版本直接装入真实训练 pipeline，**前向数值看似一致**，但梯度在小尺度上出现偏差；并且这种偏差不会触发 NaN 或显式断言，**只在多步累积后导致 loss 曲线缓慢偏离基线**——这是最危险的情况：跑得快、看不到错，但模型悄悄学坏。

类似情况在推理 kernel 中同样出现：精度退化、边缘 shape 的越界读、warp 内规约的非确定性求和顺序——这些问题在单元测试中往往因为输入分布有限而漏检，到大规模生产环境才显露。

## 评测体系的缺陷

帖子隐含的批评是：**当前 AI-for-systems 基准更接近\"刷分游戏\"而非工程验证**。SOL-ExecBench 提供了执行正确性检查，但精度容差、数值稳定性、确定性、与上下游算子的耦合行为并未被充分覆盖。AI 生成器（多为 LLM + RL）善于探索极致优化路径——例如改写规约策略、降低中间精度、利用未公开的硬件行为——这些往往以\"在容差内\"为代价，对训练而言长期累积可能是致命的。

## 对从业者的意义

1. **不要把 AI 生成的 CUDA kernel 直接放入训练**：必须用真实模型在更长的迭代窗口对比 loss/梯度统计，而不是只比对一两个 batch 的输出。
2. **建立\"数值回归\"测试集**：除了正确性之外，关注分布尾部、混合精度下的累计误差与确定性。
3. **基准设计需要进化**：执行时间不再是单一目标，**数值健康**与**长尾稳定性**应纳入排行榜。
4. **可观测性优先**：训练系统需要在线监控梯度/激活的分布漂移，以便在模型\"学坏\"前发现 kernel 引入的隐患。

## 评论与延伸

讨论区不少 ML 工程师表示有同感：在自家训练栈中把社区开源算子替换为 AI 生成版本后，曾出现\"评测全过、上线翻车\"的情况。也有研究者指出，应推动\"differential testing\" 工具链：在大量随机分布上将候选 kernel 与参考实现做端到端 ε-等价对比，并把训练步态/收敛曲线也作为验证维度。

## 结论

这则贴子是给\"用 AI 生成系统级代码\"潮流泼的一盆冷水。**评测分数好不代表能用**，尤其是在数值敏感的训练栈中。要让 AI 真正成为可信赖的 kernel 开发者，**评测、容差、数值回归、可观测性**这四件套缺一不可，否则\"静默退化\"将比明显崩溃更具破坏性。
