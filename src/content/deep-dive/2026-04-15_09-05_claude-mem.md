# 05 · thedotmack/claude-mem

> ⭐ 8,742 stars / week | 55,803 total | 2026-04-15

## a. 定位与痛点剖析

**claude-mem** 是 Claude Code 的持久化记忆插件，核心价值主张：让 Claude Code 记住它做过的所有事情，并在未来的会话中智能注入相关上下文。

这解决了 Claude Code 最被开发者诟病的问题：**会话无记忆**。每次新开一个 Claude Code 会话，它对之前做了什么、项目的坑在哪里、你的编码偏好一无所知，开发者不得不反复解释同样的背景。

更具体的痛点场景：
- 上周花3小时解决的 webpack 配置问题，这周 Claude 又建议了同样的错误方案
- 项目内的架构决策（"我们不用 Redux，用 Zustand"）每次会话都要重新声明
- 调试过程中积累的"这个 API 会返回 null"等经验无法跨会话传承

claude-mem 通过 Claude Code 的 Lifecycle Hooks 机制，在后台无感知地捕获所有工具使用观察结果，用 AI 压缩成语义摘要，并在未来的会话开始时注入相关上下文。

## b. 核心架构与技术细节

**技术栈**：TypeScript/Node.js（18+）、Bun（进程管理）、SQLite（数据存储）、Chroma（向量数据库）、Python（uv 管理）

**5层核心组件**：

1. **7个 Lifecycle Hooks**：
   - `SessionStart`：注入历史上下文
   - `UserPromptSubmit`：捕获用户输入
   - `PostToolUse`：记录工具执行结果（核心数据来源）
   - `Stop/SessionEnd`：触发摘要生成

2. **Worker Service**（Bun 管理，端口 37777）：
   - HTTP API：10个搜索端点
   - Web UI：实时记忆流可视化
   - 引用系统：每条观察有唯一 ID，可通过 `/api/observation/{id}` 访问

3. **双数据库架构**：
   - SQLite：会话、观察、摘要的关系型存储
   - Chroma Vector DB：混合语义搜索（向量+关键词）

4. **3层检索工作流（MCP Tools）**：
   - `search`：返回紧凑索引（~50-100 tokens/条）
   - `timeline`：获取特定观察的时间序列上下文
   - `get_observations`：按 ID 批量获取完整详情（~500-1000 tokens/条）
   - 约 10x token 节省（先过滤再详细获取）

5. **Privacy Control**：通过标签排除敏感内容；Progressive Disclosure 策略控制注入量

**安装**：`npx claude-mem install`（一键，自动安装依赖）

## c. 竞品对比与生态站位

| 方案 | 持久化方式 | 搜索能力 | token效率 | 自动化程度 |

## d. 开发者反馈与局限性

**正面**：
- 55k total stars 在这类工具中非常高，说明真实采用率强
- 支持 Gemini CLI（`--ide gemini-cli`），生态扩展性好
- OpenClaw 集成支持，Real-time 观察流推送到 Telegram/Discord/Slack
- Endless Mode（Beta）：仿生记忆架构用于超长会话

**局限性**：
- 依赖链复杂（Node.js + Bun + Python + Chroma），首次安装可能遇到环境问题
- AGPL-3.0 协议（注意：`ragtime/` 目录是 PolyForm Noncommercial License）——商业化限制
- Chroma 向量数据库增加了额外资源占用
- AI 摘要质量依赖 Claude 模型，弱模型下摘要质量下降会影响检索效果
- 与项目关联了 $CMEM Solana 代币（第三方发行但作者背书），引发社区争议
