## 事件背景

**Browse.sh** 由浏览器基础设施明星公司 **Browserbase** 推出,在 Product Hunt 一经登场便冲到 326 票。它的 slogan 极具画面感——"Give your agents muscle memory for automating the web"——给你的 Agent 一双能在网页上"形成肌肉记忆"的手。痛点不难理解:当下任何让 LLM 操作浏览器的方案 (Browser Use、Playwright Agent、AutoGPT-web) 都面对同一个尴尬——每接一个新网站,Agent 都要重新摸索登录、定位元素、绕过 popup、处理分页、应对 anti-bot,失败率高、Token 烧得猛。browse.sh 的提案是建立一个**开放、可共享的"浏览器自动化 SKILL"目录**,把社区已经踩过的坑沉淀成可复用的 `SKILL.md` recipes,任意 Agent 用 `browse` CLI 一行命令就能装上 "怎么登 Notion / 怎么在 LinkedIn 发帖 / 怎么导出 Stripe 报表"这种成熟动作。

## 核心机制

产品由两部分组成。① **开放目录 (catalog)**:任何开发者都可以为某个网站发布一份 SKILL.md,文件里包含目标网站 URL、所需权限、操作步骤、选择器/A11y tree 提示、断言条件以及 fallback 路径,本质是把人工总结的"操作 SOP"编成机读规约。② **browse CLI**:让 Agent 框架 (LangChain、AutoGen、Claude Code、Cursor MCP 等) 一键拉取并执行这些 skill,执行环境跑在 Browserbase 托管的远程浏览器 (亦支持本地 Playwright)。卖点有三:**可移植**——同一份 SKILL.md 在多种 Agent 框架可用;**可治理**——版本化、可审查、可签名,降低恶意脚本风险;**可复用**——避免每个团队为同一个 SaaS 重写一遍 login 代码。这套思路明显借鉴了 Anthropic Skills 和 npm 包管理器的设计哲学。

## 社区热议与争议点

讨论集中在三个角度。**第一,与 Browser Use / Firecrawl / Stagehand 的关系**:很多评论者指出 browse.sh 不是要再造一个底层框架,而是上层"内容层",和现有 runtime 互补。**第二,信任与安全**:开放目录天然带来"任何人都可发布"的风险,如果某 skill 偷偷把用户的 Cookie 外发就是灾难,社区在催 Browserbase 给出签名机制、审计流程、沙盒隔离方案。**第三,网站方反应**:LinkedIn、Cloudflare 等长期与自动化对抗的平台是否会更激进地封 IP/指纹?有人提出 SKILL 应该附带 robots-friendly 标签,只在网站允许的范围内运作。

## 行业影响与未来展望

如果 browse.sh 真的能形成"Web 自动化 npm 仓库"般的网络效应,它有可能成为 Agent 时代的关键基础设施层——LLM 不再硬靠 vision + DOM 摸索,而是借助标准化 skill 直奔目标。对 Browserbase 自身,它从 IaaS (托管浏览器) 顺势上探到 PaaS (技能市场),粘性大幅增强。对开发者生态,它有望与 Anthropic Skills、MCP server 形成"Skill 即数据,MCP 即调用"的协同格局,加速 Agent 进入生产环境。
