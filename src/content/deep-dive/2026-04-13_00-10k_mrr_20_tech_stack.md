# 每月两万美金营收，只花 20 美元技术栈——小众开发者反击VC的极简生存术

## 事件背景

这篇文章来自 Steve Hanov 的个人博客。他运营着多个每月经常性收入（MRR）超过 1 万美元的小型软件公司，却始终保持着每月仅 20 美元的技术支出。核心诉求非常明确：在互联网创业被云计算、风险投资和大规模团队叙事主导的当下，他试图证明“一个人、几台低价服务器、极简技术栈”依然可以支撑起健康盈利的独立事业。他坦言曾被投资人拒绝，因为对方质疑：“你连融资都不需要，那我们要投你什么？”

## 核心观点/产品机制

Hanov 详细拆解了自己的 exact stack（精确技术栈）：

- **服务器**：单台 Linode 或 DigitalOcean VPS，月租 5–10 美元。
- **后端语言**：Go，静态编译成单二进制文件，通过 `scp` 直接上传到服务器运行。
- **数据库**：SQLite 并开启 WAL（Write-Ahead Logging）模式，他认为本地 SQLite 访问速度“比走 TCP 到远程 Postgres 快几个数量级”。
- **本地 AI 批量任务**：在二手 RTX 3090 上运行 VLLM / Ollama， upfront 硬件成本后运行成本为零。
- **快速/智能 LLM API**：OpenRouter，按需付费，避免管理多个供应商账户。
- **AI 编程助手**：GitHub Copilot Chat，利用其按请求定价（而非按 token）的机制，用极低成本运行长 agent 会话。

他的几个实际产品包括：websequencediagrams.com、eh-trade.ca、rhymebrain.com、zwibbler.com 等。核心主张包括三点：

1. “保持近乎为零的成本，能让你获得与拿到百万美元融资但烧钱速度极快的人完全相同的跑道。”
2. 单机部署胜过分布式云复杂架构：“你准确地知道日志在哪、为什么崩溃、如何重启。”
3. Go 的静态二进制消除了依赖地狱：“在你的笔记本上编译成单个静态链接二进制，scp 到 5 美元服务器上运行即可。”

他还开源了两个自用工具：**laconic**（针对 8K 上下文优化的 agent 研究者）和 **llmhub**（本地/云端模型的 LLM 抽象层）。

## 社区热议与争议点

Hacker News 的讨论热度极高，核心争议围绕 SQLite 能否支撑商业级应用，以及“单节点万岁”是否为幸存者偏差。

**SQLite vs. Postgres 性能与架构之争**

- **hackingonempty** 为 SQLite 辩护：“本地 SQLite 通过 C 接口或内存通信，比向远程 Postgres 走一次 TCP 网络跳要快几个数量级。”
- 有人反驳称：“你可以通过 Unix domain socket 连接本机 Postgres，几乎消除所有开销。”
- **andersmurphy** 进一步加码："Even with domain sockets, SQLite smokes Postgres on the same machine."

**单节点限制与高可用批评**

- **locknitpicker** 批评："SQLite on the same machine is akin to calling fwrite... forces a one-database-per-instance design, with no data shared across nodes." 他还指出："Backups don't buy you high availability. At best, they buy you disaster recovery."
- **andersmurphy** 回击："You can scale incredibly far on a single node and have much better up time than github or anthropic." 并认为流式复制 backups 恰恰是实现高可用的关键。

**“云复杂度即习得性无助”**

- **senko** 提出更尖锐的批评：人们动不动就抛出“但扩展怎么办”“但高可用怎么办”的论点，本质上只是在重复各种云平台的推销话术，是一种“习得性无助”（learned helplessness）。
- **deaux** 用亲身经历支持：某初创公司明明可以完全跑在单台 VPS 上，却雇了专职基础设施人员搭建完整 k8s。

## 行业影响与未来展望

这篇文章切中了当前技术行业一个长期被忽视的暗流：随着云计算账单不断膨胀，以及对“增长优先、盈利次后”模式的幻灭，越来越多的开发者正在重新审视“小型软件”（small software）和“自给自足创业”（bootstrapped tech）的可行性。Hanov 的实践证明，AI 工具进一步放大了个人开发者的效率上限：一个拥有本地 GPU 和廉价 API 的程序员，其产出可能超过传统意义上的小型团队。

长远来看，这种模式可能会催生更多“一人独角兽”——未必真要冲击独角兽估值，而是以极低成本维持稳定现金流、拒绝过度融资的生活方式公司。然而，这种模式的可扩展天花板和单点故障风险，也注定让它更适合内容驱动、轻交互、低并发的利基产品，而非高并发、强一致性的企业服务。
