## 一、事件背景

r/LocalLLaMA 出现一篇关于 Qwen 3.6-27B 在 DeepSWE / SWE-bench 类基准上的实测分享。Qwen 3.6-27B 是阿里 2026 年 4 月发布的稠密开源模型（Apache 2.0），原生 262K 上下文、可扩 1M，定位代码 agent 与仓库级推理。社区此前已注意到一份 GitHub 讨论：Qwen3.6-27B-FP8 在工程化 agent 栈下跑出 SWE-bench Verified 88–90% 的成绩，超越多款体量更大的闭源模型。该帖把视角搬到 DeepSWE 这一更偏多步骤工程任务的轨道上。

## 二、核心观点

- 27B 稠密 + 长上下文，让单卡/单 DGX Spark 跑全仓库级 PR 修复成为可能，避免 MoE 路由抖动。
- FP8 部署下吞吐显著优于 FP16，配合 sglang/vLLM 在 agent 长尾任务里更经济。
- 新引入的 Thinking Preservation 机制（在多轮工具调用之间保留 reasoning 上下文）被认为是 DeepSWE 分数提升的关键。
- 27B 在工程化 agent 栈加持下能与 397B 量级闭源模型同台竞争，"模型 ≠ 全部"，agent stack 设计权重正在变大。

## 三、社区热议与争议点

1. **分数水分质疑**：有人指出 88–90% 依赖大量重试与代理层（~47K LoC 闭源 proxy），"裸模型"分数仍需独立复现；楼主提供了 CLI-swap ablation 试图反驳"任务特化作弊"。
2. **27B vs 35B-A3B MoE**：NVIDIA 论坛实测显示 35B-A3B 在某些子集更快但准确率略低；本地用户在"显存够不够 + 速度"之间分裂。
3. **FP8 量化损失**：部分用户反映 FP8 在长链 reasoning 上偶发 token 漂移，建议关键步骤回退 BF16。
4. **开源 agent stack 缺位**：最被吐槽的是 proxy 闭源，社区呼吁拆出最小可复现 harness，否则"分数归 stack，不归模型"。

## 四、行业影响与未来展望

Qwen 3.6-27B 进一步坐实了"中等稠密模型 + 强 agent harness"路线的可行性，对开源生态意义重大：它把 SWE-bench 顶尖区从"必须用闭源大模型"拉回到单机可部署区间。下一步看点在于：(1) Qwen3-Coder-Next 与 3.6-27B 在 DeepSWE 这类更真实工程基准的对垒；(2) 开源 agent stack（OpenHands、SWE-agent、Aider）能否把这套 proxy 能力公共化；(3) FP8/INT4 在 reasoning preservation 下的稳健性研究。对企业而言，自托管 27B + 私有代码库的"低成本 90 分方案"正在变成现实可选项。
