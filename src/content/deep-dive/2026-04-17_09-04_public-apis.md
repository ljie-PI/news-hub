# public-apis 深度报告

> ⭐ 424,185 stars (+1,295 today) | 维护者：public-apis org | GitHub 历史最多 star 项目之一

---

## 定位与痛点剖析

**public-apis** 是 GitHub 上 star 数最多的项目之一，其定位极其简单：**一个免费公开 API 的集合列表**，以 Markdown 格式维护在 README 中，覆盖数百个分类（动物、天气、地理、金融、AI、社交等），每条记录包含 API 名称、描述、Auth 方式（无/apiKey/OAuth）、HTTPS 支持状态、CORS 支持等字段。

核心痛点它解决的：
- 开发者找免费 API 需要大量搜索和踩坑（注册、限流、收费陷阱）
- 原型开发阶段需要快速找到合适的数据源
- 学习 API 调用时需要低门槛可用的接口

项目之所以持续被关注，是因为**它是懒人的第一站**——找 API 先去这里看看。这种"目录类"项目有独特的网络效应：越多人用 → 越多人贡献维护 → 质量越高。

---

## 核心架构与技术细节

项目本质是一个**精心维护的 Markdown 数据集**，技术含量不高但工程要求不低：

**数据结构（每条 API 记录）：**
```
| API | Description | Auth | HTTPS | CORS |
```

**验证基础设施：**
- GitHub Actions 自动校验链接可用性
- PR 提交需遵循贡献指南（格式、分类、必填字段）
- 社区审核机制防止滥用

**配套生态：**
- 官方 JSON API（`public-api-lists` 同类项目提供）
- 多个非官方的 API wrapper 和搜索界面
- 被数百个教程、Awesome 系列引用

当前约收录 **1500+ 个 API**，分 40+ 个大类。由于是纯 Markdown，无法提供机器友好的搜索——这是一个长期被社区吐槽的问题。

## 开发者反馈与局限性

**正面评价：**
- 社区贡献活跃，每周有大量 PR
- 分类清晰，HTTPS/CORS/Auth 字段对开发者决策很有用
- "第一个 open source 贡献"的新手友好型项目

**局限性（被反复讨论的问题）：**
- **Markdown 不可搜索**：找一个特定 API 需要 Ctrl+F，无法按多条件过滤
- **数据时效性差**：部分 API 已关闭或变更，但维护者无法及时发现
- **无质量分层**：好用的 API 和几乎无人维护的 API 平等排列
- **维护者疲劳**：大量 PR 审核依赖少数核心维护者，曾出现维护停滞
- **star 虚高问题**：很多人 star 但实际用到特定 API 的比例低，"收藏夹"效应
- 无法判断 API 的速率限制、文档质量、稳定性等实际使用关键因素
