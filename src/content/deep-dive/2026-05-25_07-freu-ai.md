# Freu AI：用「AOT 编译」把 Mac 桌面自动化做成 $0 运行成本

## 1. 事件背景

桌面 RPA 长期面临两难：AppleScript / 坐标点击器在窗口 resize 或 UI 改版后就崩；新一代视觉 Agent（云端 LLM + 截图）虽然「通用」，但成本与延迟过高——一个跨 5 步、每天跑 10 次的简单工作流就可能消耗 50 万 token/天。**Freu AI**（freu.ai，作者 Charles）2026-05-24 在 Product Hunt 发布，宣称要把任何 Mac 应用通过自然语言自动化，并把「日常运行成本压到 $0」。

## 2. 核心观点 / 产品机制

Freu 的核心思路是 **AOT（Ahead-of-Time）编译 + Semantic UI（SUI）**：用户演示一次工作流，云端视觉模型把它「编译」成一段确定性 DSL；之后所有重复执行都在 **本地** 直接解释 DSL，不再回云端。SUI 引擎做三步——Perceive（识别按钮/输入框/图标）、Resolve（锚定语义而非坐标，比如「Spotify 的 Play 按钮」即使位置变化仍能找到）、Execute（将视觉锚点绑入 DSL 确定性执行）。配套开源了 **freu-cli**（DOM 浏览器自动化引擎，GitHub: freu-ai/freu-cli），定位为可嵌入其它 Agent 的「web 任务肌肉记忆」。后续路线图是 **端侧 Vision 执行引擎**，让初次「学习」也无需云端。

## 3. 社区热议与争议点

社区好评集中在「teach-once，本地重放」这一定位上：Abhishek Srivastava 称「方法很简单但很酷，期待本地 vision 引擎」；Büşra Şeker 看好该模式，但同时点出最尖锐的问题——**App 升级、UI 微调后 DSL 如何自动迁移？** 这是 AOT 路线的根本风险：确定性 DSL 在 UI 漂移时会比纯视觉 Agent 更脆；Freu 的回答是依赖 Semantic UI 锚定语义而非坐标，但实际鲁棒性仍待验证。另一争议是：把工作流编译过程发到云端是否会带来隐私顾虑——Freu 承诺日常执行 100% 本地、屏幕数据不出端，以此作为对照 OpenAI Operator / Anthropic Computer Use 的差异化卖点。

## 4. 行业影响与未来展望

Freu 代表 Computer-Use Agent 赛道里的「成本派」分支：与 OpenAI Operator、Anthropic Computer Use 的「每次调用都过云端 + 高 token 成本」路线形成鲜明对比。如果 AOT 思路成立，桌面自动化的经济模型会被改写——重复任务边际成本趋零，模型只在「教学」阶段消耗。短期看，Freu 的关键考验是 SUI 在版本升级下的稳定性、以及非英语 / 非主流 App 的覆盖度；长期看，「学一次、本地跑」可能成为 personal automation 的默认形态，挤压传统 RPA 与纯云端 Agent 的中间地带。
