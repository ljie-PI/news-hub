# CSSQuake

## 1. 事件背景

CSSQuake 是一个把 1996 年经典 FPS《Quake》搬进浏览器的实验项目，HN 条目链接指向 `https://cssquake.com/`，新闻监控源记录其热度约 433 分、89 条评论；我通过 HN API 复核时，条目作者显示为 `msalsas`，分数约 435，评论数为 89，发布时间为 2026-06-20 10:49:53 UTC。项目页面本身内容很“游戏化”：主菜单包含 Multiplayer、Select Level、Help、Options，操作说明为 WASD 移动、鼠标视角、点击开火、空格跳跃、Shift 奔跑、Ctrl 蹲伏。它迅速引发讨论，并不是因为“又一个网页版 Quake”本身罕见，而是因为它把一个通常应由 WebGL/Canvas/GPU 图形管线承担的 3D 场景，刻意放进了 HTML/CSS 的渲染体系里。

## 2. 核心观点/产品机制

CSSQuake 的核心卖点是：使用 PolyCSS 将 Quake 世界渲染为真实 DOM/CSS 3D 几何，而不是用 WebGL 或把整幅画面画进 `<canvas>`。PolyCSS 官方说明称，每个多边形都会成为一个真实 DOM 元素，通过 `transform: matrix3d(...)` 定位；UV 纹理三角形会被打包为 atlas sprite，再通过 CSS background 定位，最终由浏览器 compositor 负责 3D 层叠。

GitHub 仓库 `LayoutitStudio/cssQuake` 对其架构描述更具体：项目会预处理原始 Quake 数据，生成浏览器可直接加载的 JSON、图片资产和 PolyCSS render bundles；运行时由 TypeScript 负责游戏循环、玩家移动、碰撞、敌人状态、拾取物、武器、UI、音频、路由、调试钩子以及 DOM 更新。也就是说，它不是在浏览器里运行一个完整 Quake VM；浏览器运行期也不会解析原始 PAK 或 BSP 文件，而是在构建阶段把 BSP、WAD、MDL、LMP、实体数据、可见性、碰撞、HUD、菜单、武器和 QuakeC 派生事实等准备好。`pnpm build` 只构建 Vite 应用，`pnpm build:full` 才会重新生成 Quake 资产并构建应用；仓库还支持按地图或模型做局部准备，例如 `pnpm prepare:quake:map e1m7`、`pnpm prepare:quake:model dog`。

从产品意义看，CSSQuake 的重点不在于替代成熟 3D 引擎，而是证明 Web 平台中“样式层”已经足够强大，甚至可以被滥用成一个可交互 3D 游戏渲染器。这个项目的荒诞性恰恰是吸引力：它把 CSS 的边界推到一个普通前端开发者很少会认真尝试的位置。

## 3. 社区热议与争议点

HN 讨论主要围绕性能、真实性、CSS/JS 边界和项目原创性展开。几个具体例子如下：

1. 用户 `jedberg` 赞叹项目成就，但调侃说 90 年代的 Pentium-133 PC 上 Quake 都比它在 Mac M1 Pro 上跑得顺；回复里有人指出“这是 CSS，不能期待性能优化”，也有人反馈 Chrome on M2 很流畅、Safari/WebKit 更卡，另有 Linux/Firefox 用户表示没有问题。这说明瓶颈并非单纯硬件，而与浏览器引擎对 CSS 3D transform、DOM 合成和层管理的实现密切相关。

2. 用户 `badsectoracula` 追问这是否只是 CSS 渲染器，还是完整复刻了引擎和逻辑，并举例称某些按钮、暗门行为与原版不同。项目相关回复解释说，构建阶段会从 QuakeC/progs.dat 中抽取状态、模型、攻击、声音等 JSON “source facts”，浏览器运行时则由 TypeScript 消费这些事实实现“Quake-ish”的玩法。这一讨论揭示了 CSSQuake 的真实性边界：它追求可玩和视觉复现，但不是逐指令、逐逻辑完全模拟原版。

3. 用户 `remix2000` 指出“这个 CSS Quake 似乎还是需要 JS 才能运行”。回复澄清：CSS 负责渲染，游戏逻辑是 TypeScript。这是一个重要误区——“CSSQuake”并不意味着完全不用 JavaScript，而是把世界渲染这件事交给 DOM/CSS，而输入、物理、状态机和资源调度仍然需要脚本。

4. 用户 `edwinjm` 质疑它是否借鉴或复制了 Niels Leenheer 的 cssDOOM；其他评论者指出这是独立项目，并给出源码仓库 `https://github.com/LayoutitStudio/cssQuake`。此外，还有人问是否没有声音、有人打趣“能跑 Crysis 吗”、有人说“比 vim 还难退出”，这些反应共同构成了典型 HN 氛围：一半认真审视技术边界，一半享受黑客式玩笑。

## 4. 行业影响与未来展望

CSSQuake 对产业的直接商业价值有限：没有严肃团队会为了 3D 游戏性能选择用大量 DOM 节点和 CSS transform 替代 WebGL、WebGPU 或原生引擎。但它的间接价值很高。第一，它是一次浏览器渲染管线的压力测试，能暴露 Chrome、Firefox、Safari/WebKit 在 CSS 3D 合成路径上的性能差异。第二，它是一个教育型样例，帮助开发者理解 DOM、CSS transform、纹理切片、资产预处理和游戏运行时之间的关系。第三，它也说明“前端技术栈的非预期用途”仍然有创新空间：当每个多边形都可被 DOM 选择器命中、添加事件、应用类名和动画时，3D 内容与传统网页交互模型之间可能产生新的混合形态。

未来这类项目更可能走向三个方向：一是作为 PolyCSS 的展示项目，继续完善模型支持、光照、阴影和调试工具；二是作为复古游戏移植/浏览器考古的一部分，探索 CSS、SVG、Web Components 等非传统渲染路径；三是反过来推动浏览器厂商关注复杂 CSS 3D 场景的退化路径和调试体验。CSSQuake 不会成为主流游戏引擎，但会成为一个很好的“边界案例”：当有人问 CSS 能不能做 3D FPS，答案不再只是理论上的“不该做”，而是“可以，甚至能玩，但你要接受它的荒诞和代价”。
