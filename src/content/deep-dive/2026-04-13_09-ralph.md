## 定位与痛点剖析

当你用 AI 编程工具（Amp、Claude Code）构建复杂功能时，会遇到一个普遍问题：**单次 AI 会话的上下文窗口有限，无法完成跨越多个文件、多个步骤的复杂任务**。更大的挑战是，即使一次能完成，AI 也可能在"完成"一个功能后就停下来等待人类的下一步指令，导致开发效率的天花板始终受限于人类的响应速度。

`ralph` 的定位是：**一个让 AI 自主、循环地完成 PRD 中所有需求项的 bash 执行框架**。它的名字来自 Geoffrey Huntley 提出的 "Ralph Pattern"——一种将产品需求文档（PRD）结构化为 JSON 任务列表，然后用脚本驱动 AI 一条一条完成直到全部通过的开发模式。

核心价值主张：**把 AI 从"响应工具"升级为"执行引擎"**，让开发者可以描述 What，AI 负责完成 How，直到所有验收标准全部满足。

## 核心架构与技术细节

ralph 本质上是一个约 100 行的 **bash 脚本**（`ralph.sh`），但其设计思想极为精妙。

### 核心循环逻辑（ralph.sh）

```
while 存在 passes=false 的 story AND 未达到最大迭代次数:
    1. 从 prd.json 中挑选最高优先级的未完成 story
    2. 启动新的 AI 实例（Amp 或 Claude Code）
    3. AI 实例完成该 story（包括编码、typecheck、测试）
    4. 若所有检查通过 → git commit，更新 prd.json 中该 story 的 passes=true
    5. 将本次学习追加到 progress.txt
    6. 销毁当前 AI 实例，进入下一轮循环
当所有 story passes=true → 输出 <promise>COMPLETE</promise>，循环退出
```

### 关键设计原则

**每次迭代 = 全新 AI 实例，干净上下文**

这是 ralph 最反直觉但最关键的设计决策。每轮循环都 spawn 一个**全新的 AI 实例**，没有前轮的上下文污染。跨迭代的"记忆"通过三种机制传递：
1. **Git history**：已提交的代码即已完成的事实
2. **progress.txt**：append-only 的学习日志，每轮结束后追加发现的 pattern、坑点
3. **prd.json**：每个 user story 带 `passes` 字段，是任务状态的唯一真源

**AGENTS.md 更新机制**

每轮迭代后，AI 被要求更新项目中的 `AGENTS.md` 文件，记录：
- 发现的代码约定（"这个项目用 X 做 Y"）
- 需要注意的坑（"修改 W 时别忘了更新 Z"）
- 有用的上下文（"设置面板在组件 X 里"）

因为 Amp 和 Claude Code 会自动读取 `AGENTS.md`，这些更新会被未来的所有迭代自动继承。

**PRD 格式（prd.json）**

```json
{
  "branchName": "feature-user-auth",
  "userStories": [
    {
      "id": "1",
      "title": "Add login form",
      "description": "...",
      "acceptanceCriteria": ["...", "Verify in browser using dev-browser skill"],
      "passes": false
    }
  ]
}
```

**Story 粒度控制**：每个 story 必须小到能在单个上下文窗口内完成。文档明确给出反例：
- ❌ "构建整个 dashboard"（太大）
- ✅ "给列表页加一个筛选下拉框"（合适）

**支持的 AI 工具**：
- `amp`（默认）—— Sourcegraph 的 Amp CLI
- `claude`（`--tool claude`）—— Anthropic Claude Code

**Amp 自动切换（auto-handoff）**：
```json
{"amp.experimental.autoHandoff": {"context": 90}}
```
当上下文使用率达到 90%，Amp 自动切换新实例，ralph 无缝继续——这解决了超大 story 的上下文溢出问题。

### 反馈回路

ralph 只在有"反馈回路"时才能可靠工作：
- TypeScript typecheck 捕获类型错误
- 单元/集成测试验证行为
- CI 必须保持绿色（broken 代码会跨迭代累积）
- 前端 story 需要用 `dev-browser` skill 在浏览器中验证

## 竞品对比与生态站位

| 方案 | 方式 | 人工介入 | 适用规模 |
|------|------|---------|---------|
| **ralph** | bash 循环 + prd.json | 极少 | 中等功能开发 |
| **手动迭代 Claude Code** | 人工触发 | 高 | 任意 |
| **Devin** | 完整 AI 软件工程师 | 极少 | 大型任务（SaaS） |
| **SWE-agent** | Academic agent loop | 中等 | 研究/基准测试 |
| **OpenHands** | 完整 Agent 框架 | 中等 | 通用任务 |

ralph 的定位比 Devin/OpenHands 轻得多——它不是"完整 AI 工程师"，而是"让现有 AI 工具自主执行已规划好的任务列表"的胶水层。这使它对已有 Amp/Claude Code 工作流的开发者几乎零学习成本。

## 开发者反馈与局限性

**正面**：
- 作者 Ryan Carson 发布的详细使用文章引发广泛讨论
- 对于结构良好的 PRD，自动完成率据报道相当高
- bash 脚本实现极为轻量，无需额外依赖（除 `jq`）

**局限**：
1. **依赖 PRD 质量**：prd.json 的粒度和清晰度直接决定成功率，story 写得不好会导致 AI 反复失败
2. **测试覆盖前提**：没有自动化测试，ralph 无法可靠判断 story 是否真正完成
3. **调试困难**：当某个 story 反复失败，排查需要读取 git log、progress.txt、prd.json，不够直观
4. **成本不可控**：每次迭代都是完整的 AI 调用，复杂项目的成本可能很高
5. **仅支持 Amp 和 Claude Code**：不支持 Cursor、Copilot 等其他工具
