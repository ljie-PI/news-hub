# firecrawl/firecrawl

## 定位与痛点剖析
Firecrawl 是一套面向 AI 应用的 Web 数据 API，把「搜索、抓取、与网页交互」三件事统一成云端接口。传统爬虫需要自己处理代理轮换、限流、JS 渲染与反爬，输出还得再清洗成模型可读的格式。Firecrawl 直接返回干净的 Markdown、结构化 JSON 或截图，主打「零配置」拿到 LLM-ready 数据。它面向搭建 RAG、Agent、数据管道的开发者，既开源自托管，也提供托管服务。README 自报口径称覆盖 96% 网页、P95 延迟 3.4 秒，这些为官方基准，非独立第三方结论。

## 核心架构与技术细节
主体以 TypeScript 编写，采用 AGPL-3.0 许可。核心端点包括 Search、Scrape、Crawl、Map、Batch Scrape，以及新增的 Agent 与 Interact。Agent 是原 /extract 的演进，可根据自然语言描述自动搜索并抓取，无需预先提供 URL，并支持 Pydantic schema 输出结构化数据；模型分 spark-1-mini 与 spark-1-pro 两档。Interact 支持在抓取后对页面执行点击、滚动、输入等动作。项目还提供 CLI 与 firecrawl-mcp，可一键接入 Claude Code、OpenCode 等 MCP 客户端。

## 竞品对比与生态站位
README 在模型对比示例里直接点名 Apify 与 ScrapingBee 作为参照，另有 Jina Reader、Crawl4AI 等同类。相比自建 Playwright 方案，Firecrawl 省去代理与编排的运维成本；相比纯开源库，它以托管 API 和 MCP 生态取胜，深度绑定 Agent 工作流。生态位处在「网页到 LLM 上下文」这一层，是 AI 应用的数据入口。

## 开发者反馈与局限性
仓库当前约 388 个 open issues。近期 issue 与 PR 集中在企业级安全能力，如 threat protection 域名风险拦截、IP allowlist、per-key 权限锁定，反映其正向商业化演进。也暴露解析瑕疵，如 /v2/parse 处理 DOCX 时相邻加粗片段被 Word 格式化拆词，以及需升级 golang.org/x/net 修复 CVE。核心引擎依赖托管服务与 API key，自托管完整能力门槛较高。
