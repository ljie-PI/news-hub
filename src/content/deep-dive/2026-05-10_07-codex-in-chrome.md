# Codex in Chrome 深度解析

## 一句话总结

OpenAI 于 2026 年 5 月 7 日正式推出 **Codex Chrome 扩展**，让 Codex 能够直接操作用户已登录的 Chrome 浏览器，完成跨标签页信息收集、网页应用测试、表单填写、DevTools 调试等任务，标志着 AI 编程助手从“代码生成”迈向“端到端浏览器自动化”。

---

## 核心功能

| 能力 | 说明 |
|------|------|
| **认证会话操作** | 在已登录的网站（LinkedIn、Salesforce、Gmail、内部工具）中自动执行任务 |
| **跨标签页上下文** | 收集并整合多个打开标签页的信息 |
| **Chrome DevTools 集成** | 与开发者工具交互，辅助调试 |
| **多标签测试** | 同时测试多个网页应用 |
| **后台任务执行** | 尽量不影响用户当前活跃会话 |
| **智能切换** | 自动在插件、Chrome 扩展、Codex 内置浏览器之间选择最佳工具 |

---

## 安装与使用

1. 在 Codex 应用中进入 **Plugins**，添加 **Chrome** 插件；
2. 按引导安装 Chrome 扩展并授权；
3. 在任意新线程中直接调用：
   ```markdown
   @Chrome open Salesforce and update the account from these call notes.
   ```
4. Codex 会自动按线程组织 Chrome 标签页组（tab groups）。

## 安全与权限模型

### 网站访问控制
- **默认行为**：首次访问新域名时，Codex 会主动询问用户；
- **选项**：允许一次 / 始终允许该域名 / 拒绝；
- **黑白名单**：可在 Computer Use 设置中管理。

### 高风险设置
| 设置 | 风险等级 | 说明 |
|------|----------|------|
| Always allow browser content | 高风险 | 不再对任何网站弹出确认 |
| Browser history access | 高风险 | 将浏览历史纳入任务上下文，无法“始终允许” |

### 扩展权限（安装时请求）
- 访问页面调试器
- 读取/修改所有网站数据
- 读取/修改所有已登录设备的浏览历史
- 显示通知、管理书签、下载、标签组等

> **重要**：Chrome 权限仅提供“能力”，Codex 自身的确认、设置、黑白名单才决定是否真正访问。

### 数据存储
- OpenAI **不会单独存储**完整的 Chrome 操作记录；
- 仅当浏览器活动成为 Codex 上下文（页面文本、截图、工具调用、摘要、消息）时才会存储；
- 适用 ChatGPT 与 Codex 的数据控制策略。

## 战略意义与未来方向

1. **从代码生成到浏览器代理**：Codex 不再只是写代码，而是能直接在真实浏览器环境中测试、调试、操作已登录应用；
2. **打通企业内部工具**：支持 Salesforce、Gmail、内部系统等需要身份认证的 SaaS，极大扩展了企业场景；
3. **与 ChatGPT / Atlas 浏览器整合**：Engadget 报道称 OpenAI 计划推出统一应用，整合 Codex、ChatGPT 与 Atlas 浏览器（具体时间未公布）。

---

## 适用场景

- 在已登录的 CRM/ERP 中自动更新客户信息；
- 跨多个标签页收集竞品资料并汇总；
- 自动填写复杂表单、提交审批；
- 结合 DevTools 进行前端调试与性能测试；
- 在本地开发服务器与线上环境之间切换测试。

---

## 风险与建议

- **凭证暴露**：扩展运行在用户已登录会话中，需警惕恶意或误导性页面内容；
- **敏感数据**：避免通过浏览器任务传输密码、密钥等高度敏感信息；
- **企业治理**：团队应将扩展视为自动化运行时，验证权限边界与审计轨迹后再用于生产账号。

---

## 结论

Codex in Chrome 是 OpenAI 将 AI 编程助手从“IDE 内代码生成”推向“全浏览器自动化”的关键一步。凭借 400 万周活用户和 8 倍增速，它正在快速成为开发者和知识工作者在日常 Web 工作流中的核心工具。对于需要操作认证网站、跨标签页收集信息或进行 Web 应用测试的用户而言，这款扩展显著提升了 Codex 的实用边界，但也带来了更高的安全治理要求。
