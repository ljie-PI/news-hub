## 背景与问题

r/artificial 上的一位开发者描述了一个常见但棘手的场景：他们在生产环境中部署了一个 AI agent，但用户反馈"输出不可用"。具体表现是 agent 返回的结果格式与用户预期不符——即便用户在 prompt 或对话中明确给出了格式要求，agent 仍然倾向于返回自由文本或带有解释性段落的内容。发帖人希望听听社区中其他人在发布 AI 功能后"翻车"的经历：你怎么定位问题？哪些工具和指标最关键？

这其实是 LLM 产品化最常见的故障模式之一：**模型层面"基本可用"，但产品层面"完全不可用"**。

## 为什么"格式不可用"如此普遍

1. **指令遵循的脆弱性**：即使是 GPT-4 级别的模型，遇到稍长的系统提示、多轮对话漂移、或与训练数据习惯冲突的格式（例如严格 JSON Schema、带特殊分隔符的结构）时，遵循率会显著下降。
2. **Agent 链路放大误差**：单步 90% 的格式准确率，到 5 步链式调用后只剩 ~59%。
3. **用户输入分布偏移**：开发时的样本与真实用户输入差异巨大，长尾 query 触发模型"创意发挥"。
4. **温度与采样配置不当**：很多团队默认 temperature=0.7 部署生产，结构化输出场景应该接近 0。

## 调试方法论

**第一步：建立可观测性**。任何 LLM 产品上线第一件事不是优化 prompt，而是日志。至少记录：完整 prompt、完整 response、token 数、延迟、用户后续行为（是否重试、是否复制、是否流失）。推荐工具：Langfuse、Helicone、LangSmith、Phoenix（Arize）。

**第二步：分层归因**。把"输出不可用"拆解为：(a) 格式错误（解析失败），(b) 内容错误（事实/逻辑），(c) 风格错误（语气/长度）。三类问题的解决路径完全不同。

**第三步：构建评测集**。从生产日志中抽样 100–500 条真实失败案例，标注期望输出，跑回归。每次改 prompt 或换模型都跑一次，避免"按下葫芦浮起瓢"。

## 关键工具与指标

- **结构化输出**：优先用 OpenAI structured outputs / Anthropic tool use / Outlines / Instructor 库强约束，而不是靠 prompt 祈祷。
- **关键指标**：格式合规率、JSON 解析成功率、用户重试率、人工 override 率、p95 延迟。
- **A/B 与 Shadow Mode**：新版本先 shadow 跑一周，对比真实分布上的差异，再灰度发布。

## 给发帖人的建议

针对"格式不对"这个具体症状，最快的止血方案是：(1) 切换到支持 JSON Schema 的模型 API；(2) 在 prompt 末尾加 few-shot 示例而非长描述；(3) 加一层后处理校验+重试。长期则需要建立评测+回归流水线，让模型迭代变成工程问题而非玄学。
