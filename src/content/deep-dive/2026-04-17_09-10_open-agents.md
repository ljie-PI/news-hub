# vercel-labs/open-agents 深度报告

> ⭐ 3,203 stars (+738 today) | 出品：Vercel Labs | 定位：开源云端 Agent 参考实现

---

## 定位与痛点剖析

**open-agents** 是 Vercel 官方实验室出品的**开源云端编程 Agent 参考应用**，设计为「可 fork 并按需定制」而非「黑盒平台」。

核心痛点：
- 在云端运行 coding agent 需要解决「持久化执行」问题——HTTP 请求有超时，但 agent 任务可能需要几分钟甚至更长
- 每个人从头搭 sandbox 环境（文件系统 + shell + git + preview port）成本极高
- Agent 与 sandbox 的分离（agent 在外，sandbox 在内）是一个反直觉但重要的架构决策，大多数人不会自己想到

open-agents 用三层架构回答了这个问题：
```
Web UI → Agent Workflow（持久化） → Sandbox VM
```

关键洞见是：**Agent 不运行在 sandbox 内，而是从外部通过工具与 sandbox 交互。** 这使得：
- Agent 执行生命周期独立于 HTTP 请求
- Sandbox 可以独立 hibernate / resume
- 模型/provider 可以换而不影响 sandbox

## 核心架构与技术细节

**三层系统：**

1. **Web App（`apps/web`）：** Next.js，处理 auth、sessions、chat UI、streaming
2. **Agent Workflow（`packages/agent`）：** Vercel Workflow SDK 驱动的持久化多步执行，支持 streaming 和 cancellation
3. **Sandbox VM（`packages/sandbox`）：** 基于 snapshot 的 Vercel Sandbox，暴露端口 3000/5173/4321/8000，闲置后 hibernate

**Agent 具备的工具：** 文件读写、搜索、shell 命令、task 管理、skill 调用、web 访问

**当前能力：**
- Chat 驱动的编程 Agent，可 clone repo、在 branch 上工作
- 自动 commit / push / PR 创建（可选）
- 会话分享（只读链接）
- ElevenLabs 语音输入（可选）
- 基于 snapshot 的 sandbox 快速 resume

**技术栈：** Next.js + Bun + TypeScript + Neon PostgreSQL + Upstash Redis（可选）+ Vercel Sandbox

**部署要求（最小化）：**
```env
POSTGRES_URL=
JWE_SECRET=
```
完整功能需要 Vercel OAuth App + GitHub App 配置。

## 竞品对比与生态站位

| 方案 | 云端持久化 | 开源 | Sandbox 隔离 | 一键部署 |
|------|-----------|------|-------------|---------|
| Devin | ✅ | ❌ | ✅ | ❌ |
| GitHub Copilot Workspace | ✅ | ❌ | ✅ | ❌ |
| SWE-agent | 有限 | ✅ | Docker | ❌ |
| **open-agents** | **✅** | **✅** | **✅（Vercel Sandbox）** | **✅（Vercel）** |
| E2B | ✅（SDK） | 有限 | ✅ | 需自建 |

open-agents 的差异在于**Vercel 全栈整合**：一键 Fork + Deploy，native 支持 Vercel Sandbox，对已经在 Vercel 生态的团队极其友好。

但这也是它的限制：强依赖 Vercel 平台（Workflow SDK、Sandbox、Postgres 集成），迁移到其他平台成本较高。

## 开发者反馈与局限性

**正面评价：**
- Vercel 官方出品，架构决策有参考价值（尤其是 agent/sandbox 分离哲学）
- 代码清晰，「设计为 fork，不是黑盒」的理念受工程师欢迎
- 完整的 GitHub 集成（clone/push/PR）开箱即用
- Workflow SDK 驱动的持久化执行，天然解决了 serverless 超时问题

**局限性：**
- **强 Vercel 平台绑定**：Vercel Sandbox、Workflow SDK、Neon 集成——脱离 Vercel 需大量改造
- 初始部署需要配置 5+ 个 OAuth/App（Vercel OAuth + GitHub App），对新手不友好
- Vercel Sandbox 的成本在规模化使用下可能较高
- Linux 预构建 sandbox 内的工具链较基础，复杂项目可能需要自定义 snapshot
- 社区还很新（4 天前更新），生产实践案例几乎为零
- 不支持私有化部署，所有 agent 执行依赖 Vercel 基础设施
